# Codex 수업 작업물 개인 계정 이관용 통합 프롬프트

이 문서는 삼성생명 Codex 수업 종료 후, 수강생이 지급 계정에서 작업한 산출물을 개인 Codex 계정과 개인 개발 환경으로 가져가기 위한 안내 프롬프트입니다.

핵심 원칙은 다음과 같습니다.

* Codex 대화 경험 전체는 그대로 이관되지 않습니다.
* 대신 프로젝트 목적, 작업 원칙, 현재 상태, 결정 이유, 검증 기록, 다음 작업은 Markdown 문서로 이관할 수 있습니다.
* API key, 로그인 세션, 외부 계정 연결, 회사 내부 데이터는 옮기지 않고 개인 계정에서 새로 설정합니다.

## 강사용 안내 문구

수업 마지막 15분은 "내 프로젝트 인수인계 파일 만들기" 시간입니다.

Codex 대화 경험은 그대로 옮겨지지 않지만, 오늘 만든 작업 원칙, 결정, 검증 기록, 다음 할 일은 문서로 가져갈 수 있습니다.

아래 프롬프트를 Codex에 붙여넣고, 수업 중 작업한 폴더 경로를 알려주세요. Codex가 여러 작업 폴더를 조사한 뒤 개인 계정으로 가져갈 수 있는 `\\\_handoff/` 문서 세트를 만들어줍니다.

## 수강생용 통합 프롬프트

아래 내용을 그대로 Codex에 붙여넣으세요.

````markdown
# Codex 수업 작업물 개인 계정 이관용 인수인계 문서 생성 요청

나는 삼성생명 Codex 수업 중 별도 지급된 계정에서 여러 폴더에 걸쳐 작업했습니다.
수업이 끝난 뒤 이 작업물을 내 개인 Codex 계정과 개인 개발 환경으로 가져가고 싶습니다.

목표는 지급 계정의 대화 경험을 100% 옮기는 것이 아니라, 새 Codex 계정이 내 프로젝트의 목적, 파일 구조, 작업 원칙, 현재 상태, 다음 작업을 빠르게 이해하고 이어서 작업할 수 있게 만드는 것입니다.

## 0. 먼저 확인할 것

바로 파일을 만들지 말고, 먼저 나에게 작업 폴더 후보를 물어봐 주세요.

예시 질문:

- 수업 중 작업한 폴더 경로를 알려주세요.
- 최종본이라고 생각하는 폴더가 있나요?
- 실험용, 백업용, 중간 산출물 폴더가 따로 있나요?
- 개인 계정으로 꼭 가져가고 싶은 프로젝트가 몇 개인가요?

내가 폴더 목록을 주면, 그 범위 안에서만 조사하고 문서를 만들어주세요.
확실하지 않은 폴더는 추정하지 말고 "사용자 확인 필요"로 표시해주세요.

## 1. 작업 원칙

다음 원칙을 반드시 지켜주세요.

- 원본 파일을 삭제하거나 덮어쓰지 마세요.
- 기존 프로젝트 코드를 임의로 수정하지 마세요.
- 인수인계용 Markdown 문서만 새로 만들어주세요.
- API key, password, token, secret, 실제 개인정보, 회사 내부 데이터는 문서에 절대 포함하지 마세요.
- `.env`, `.env.local`, `config.local.js`, `.vercel`, `node\\\_modules`, `.venv`, cache 파일은 이관 대상에서 제외해주세요.
- 여러 폴더가 섞여 있으면 "최종본", "실험본", "백업", "중간 산출물", "확인 필요"로 나눠주세요.
- 확인된 사실, 파일을 보고 추정한 내용, 사용자가 나중에 확인해야 할 내용을 구분해서 써주세요.
- "로컬에서 실행됨", "배포됨", "운영 가능함", "검증하지 못함"을 섞어 말하지 마세요.

## 2. 만들어야 할 위치

조사가 끝나면, 최상위 작업 폴더에 `\\\_handoff/` 폴더를 만들고 아래 Markdown 파일들을 작성해주세요.

```text
\\\_handoff/
  folder\\\_map.md
  AGENTS.md
  README.md
  handoff.md
  decisions.md
  validation\\\_log.md
  prompts.md
  migration\\\_checklist.md
```

## 3. 파일별 작성 내용

### `\\\_handoff/folder\\\_map.md`

여러 작업 폴더의 지도를 만들어주세요.

포함할 내용:

- 조사한 폴더 목록
- 각 폴더의 역할
- 최종본 후보
- 실험/백업/중간 산출물 후보
- 개인 계정으로 가져가야 할 폴더
- 가져가지 않아도 되는 폴더
- 가져가면 위험한 파일 또는 폴더
- 사용자 확인이 필요한 항목

각 항목은 다음 형식으로 정리해주세요.

```text
폴더:
역할:
상태: 최종본 / 실험본 / 백업 / 중간 산출물 / 확인 필요
근거:
이관 여부:
주의사항:
```

### `\\\_handoff/AGENTS.md`

새 Codex 계정이 이 프로젝트를 이어서 작업할 때 따라야 할 운영 지침을 작성해주세요.

포함할 내용:

- 프로젝트 목적
- 이 프로젝트에서 Codex가 우선적으로 이해해야 할 맥락
- 주요 파일 구조
- 작업할 때 지켜야 할 원칙
- 승인 없이 하면 안 되는 작업
- 개인정보/API key/회사 데이터 보안 원칙
- 검증 명령
- 완료 보고 방식
- 아직 남아 있는 위험 또는 주의사항

`AGENTS.md`는 긴 작업 회고가 아니라, 앞으로 Codex가 이 프로젝트에서 어떻게 일해야 하는지 알려주는 지시문으로 작성해주세요.

### `\\\_handoff/README.md`

사람이 개인 계정/개인 PC에서 다시 실행하기 위한 안내서를 작성해주세요.

포함할 내용:

- 프로젝트 개요
- 필요한 프로그램
- 필요한 외부 계정
- 설치 방법
- 실행 방법
- 환경변수 설정 방법
- 새로 만들어야 하는 API key 목록
- 개인 계정에서 다시 연결해야 하는 서비스
- 자주 막히는 지점
- 첫 실행 확인 방법

단, 실제 API key 값은 절대 쓰지 말고 변수명만 적어주세요.

### `\\\_handoff/handoff.md`

현재 작업 상태를 인수인계 문서로 정리해주세요.

포함할 내용:

- 지금까지 한 일
- 현재 동작하는 것
- 아직 안 되는 것
- 다음에 해야 할 일
- 개인 계정에서 처음 실행할 순서
- 새 Codex에게 먼저 시킬 일
- 사용자가 직접 확인해야 할 것

상태는 다음 중 하나로 명확히 표시해주세요.

```text
draft
staged
locally verified
deployed but not operational
deployed and operational
not verified
unknown
```

### `\\\_handoff/decisions.md`

중요한 결정과 그 이유를 정리해주세요.

포함할 내용:

- 결정한 내용
- 결정한 이유
- 대안
- 영향 범위
- 나중에 다시 검토할 조건
- 검증 상태

형식:

```text
날짜:
결정:
이유:
대안:
영향 범위:
검증 상태:
재검토 조건:
```

### `\\\_handoff/validation\\\_log.md`

검증 기록을 정리해주세요.

포함할 내용:

- 실제로 실행한 명령
- 통과한 검증
- 실패한 검증
- 실행하지 못한 검증
- 실패 원인 추정
- 다음 검증 단계

명령을 실행할 수 있다면, 키나 외부 권한 없이 가능한 최소 검증만 실행해주세요.
외부 계정 로그인, 배포, DB schema 변경, 실제 데이터 적재는 하지 말고 "사용자 승인 필요"로 표시해주세요.

### `\\\_handoff/prompts.md`

수업 중 효과 있었던 프롬프트와, 개인 계정에서 이어서 쓸 수 있는 프롬프트를 정리해주세요.

포함할 내용:

- 프로젝트 재시작 프롬프트
- 코드 수정 요청 프롬프트
- 오류 해결 요청 프롬프트
- 배포 전 점검 프롬프트
- 문서 정리 프롬프트
- 다음 학습/개선 요청 프롬프트

API key, 계정 정보, 개인정보, 회사 내부 데이터가 포함된 프롬프트는 제외하거나 안전하게 익명화해주세요.

### `\\\_handoff/migration\\\_checklist.md`

개인 계정으로 옮길 때 사용할 체크리스트를 만들어주세요.

포함할 내용:

- 가져갈 파일/폴더
- 가져가면 안 되는 파일/폴더
- GitHub private repo로 옮기는 방법
- ZIP 파일로 옮기는 방법
- 개인 계정에서 새로 만들어야 하는 것
- 개인 계정에서 다시 설정해야 하는 환경변수
- 마이그레이션 후 첫 검증 명령
- 정상 이관 판단 기준
- 문제가 생겼을 때 확인할 순서

## 4. 보안 점검

문서 작성 전후로 secret이 포함될 가능성이 있는 파일을 점검해주세요.

가능하면 아래 유형을 검색해주세요.

- OpenAI/BizRouter API key
- Supabase service role key
- JWT token
- `.env` 파일
- Vercel project 연결 정보
- 실제 개인정보
- 회사 내부 데이터

문서에는 실제 값이 아니라 변수명과 설정 위치만 적어주세요.

예시:

```text
BIZROUTER\\\_API\\\_KEY=개인 계정에서 새로 발급
SUPABASE\\\_URL=개인 Supabase 프로젝트에서 확인
SUPABASE\\\_PUBLISHABLE\\\_KEY=브라우저 노출 가능한 publishable key만 사용
```

## 5. 최종 보고

문서 작성이 끝나면 마지막에 아래 형식으로 보고해주세요.

```text
1. 생성한 파일
2. 조사한 폴더
3. 최종본으로 보이는 폴더
4. 이관하면 안 되는 파일/폴더
5. 검증한 것
6. 검증하지 못한 것
7. 개인 계정에서 다음에 해야 할 일
8. 현재 상태
```

중요:
이 작업은 계정, 대화 기록, API key, 로그인 세션을 옮기는 작업이 아닙니다.
오직 프로젝트 파일과 작업 맥락을 개인 계정에서 다시 이어갈 수 있도록 인수인계 문서로 정리하는 작업입니다.
````
